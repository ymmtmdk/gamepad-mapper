## 📊 分離の基準

### ✅ プロダクトドキュメント（docs/直下）
**対象ユーザー**: 開発者・エンドユーザー・新規参入者
**目的**: 実際の開発・使用・設定に必要な情報

| ドキュメント | 用途 | 対象ユーザー |
|-------------|------|------------|
| `API_REFERENCE.md` | API使用方法・リファレンス | 開発者 |
| `DEVELOPER_GUIDE.md` | 開発環境構築・実践ガイド | 新規開発者 |
| `TROUBLESHOOTING.md` | 問題解決・トラブル対応 | 全ユーザー |
| `CONFIGURATION_GUIDE.md` | 設定ファイル詳細・カスタマイズ | ユーザー・開発者 |
| `*_ORIGINAL.md` | 詳細な技術情報（必要時のみ） | 上級開発者 |

### 🔍 内部資料（docs/internal/）
**対象ユーザー**: プロジェクトマネージャー・テクニカルリード・将来の改善作業者
**目的**: プロジェクト管理・分析・改善作業の記録・参考

| カテゴリ | 資料例 | 用途 |
|----------|--------|------|
| **分析レポート** | `DOCUMENT_IMPLEMENTATION_ANALYSIS.md` | 問題特定・改善計画立案 |
| **作業実行レポート** | `LOGGER_MIGRATION_PLAN.md` | 移行手順・成果記録 |
| **戦略文書** | `REFACTORING_STRATEGY.md` | 長期改善計画・技術方針 |

## 📚 新しい利用方法

### 👥 **新規開発者の場合**
```
1. docs/README.md でドキュメント構成を確認
2. メインドキュメント（API_REFERENCE, DEVELOPER_GUIDE）で学習
3. 必要に応じて CONFIGURATION_GUIDE で設定方法を確認
4. 問題発生時は TROUBLESHOOTING を参照
```

### 🔧 **プロジェクト改善作業の場合**
```
1. docs/internal/README.md で内部資料構成を確認
2. 分析レポートで現状・課題を把握
3. 過去の作業レポートで手法・工数を参考
4. 新しい改善作業完了後、レポートを作成・保存
```

### 📊 **プロジェクト状況確認の場合**
```
1. docs/internal/ の作業レポートで改善履歴を確認
2. MIGRATION_SUCCESS_SUMMARY で大規模改善の成果確認
3. 未完了の課題・次の改善項目を特定
```
